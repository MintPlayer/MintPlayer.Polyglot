# Build-time plugin auto-download from `pgconfig.json` (PRD)

> **GitHub:** [MintPlayer/MintPlayer.Polyglot#30](https://github.com/MintPlayer/MintPlayer.Polyglot/issues/30)
> — *"polyglot build should auto-download pgconfig.json plugin dependencies (npm) instead of requiring
> manual 'polyglot install'"*. This cashes the unbuilt half of the P10 §6 promise
> (`docs/design/plugins-and-targets.md`): *"From it the CLI resolves and **downloads** the named plugins
> into a shared cache, then emits."*

- **Status:** Draft v1.0 · 2026-07-16 · ships as a lockstep **minor** (new CLI behavior + lockfile) — the
  PR carries `release:minor`; versions stay build-injected `0.0.0-dev` placeholders in-tree (P24).
- **Author:** Pieterjan (with Claude Code).
- **Provenance:** 3-agent investigation (CLI resolution/install/cache code · design-docs + host
  integrations · npm registry protocol + zero-dep C++ implementation options), verified against source.
  Findings folded in below with `file:line`.
- **Design-of-record being implemented:** `docs/design/plugins-and-targets.md` §6/§6.1/§6.3 and
  `docs/design/json-plugins.md` §5.4 ("npm HTTP API data-only, SHA-512 `dist.integrity` verify,
  zip-slip-safe extract, never runs lifecycle scripts").

---

## 1. Problem

`pgconfig.json` declares the plugin packages a project needs (`dependencies`) and the languages to emit
(`targets`). The design promises the CLI resolves **and downloads** those plugins; on master the download
half doesn't exist. `polyglot build` can only resolve a dependency that is already local:

- an in-box plugin next to the exe (loaded at startup, `main.cpp:1434-1448`),
- a `file:<dir>` spec (`resolveConfiguredTargets`, `main.cpp:178-189`), or
- a plugin previously placed in the user cache by a **manual** `polyglot install`.

Any versioned/npm spec is rejected at build time with
`unsupported spec '…' (only file:<dir> resolves in-place; use `polyglot install …`)`.

Real-world consequence (the motivating pain): in MintPlayer.AI the `.pg` sources are supposed to
single-source a C# + TypeScript twin, but because build-time resolution can't fetch/select plugins from
config, the NuGet MSBuild integration hardcodes `--target csharp`
(`MintPlayer.Polyglot.MSBuild.targets` line 69) and the TS twins are regenerated by hand and committed —
exactly the stale-twin drift Polyglot exists to eliminate.

**Architecture intent (issue #30):** `polyglot.exe` is the single entrypoint. Every host is already a thin
launcher — the VS Code extension spawns `polyglot lsp` with no plugin logic (`editors/vscode/extension.js`
164-165), the NuGet `.targets` just `Exec`s the bundled exe. This PRD completes that picture: **all
plugin/target resolution — including downloading — lives inside the exe**, sourced from the nearest
`pgconfig.json` up the folder tree. Hosts pass files and paths, never languages.

## 2. Current reality (investigation findings)

All resolution/install/config logic lives in `src/MintPlayer.Polyglot.Cli/src/main.cpp`; Core
(`backend.cpp`/`compiler.cpp`) owns the in-memory registry + `validateBackend` and is deliberately IO-free.

- **`loadPgConfig`** (`main.cpp:112-141`) walks up the tree; `dependencies` parsed as string specs only
  (name → spec), no version-range handling, no lockfile anywhere in the repo (greenfield).
- **`resolveConfiguredTargets`** (`main.cpp:178-189`) is the single choke point — called by `runBuild`
  (`main.cpp:525`), the watch loop per cycle (`main.cpp:384`), **and** the LSP per analysis
  (`main.cpp:928`). It handles `file:` deps, errors on anything else, then probes the user cache **by bare
  target name** for each `targets` entry.
- **`runInstall`** (`main.cpp:1357-1428`) shells out to system **`npm pack`** + system **`tar`** via
  `std::system`, validates with `validateBackend`, and copies the manifest to
  `pluginCacheDir()/<manifest-name>/polyglot-plugin.json` — **no version in the path** (a second install
  overwrites the first), **no integrity check, no zip-slip guard, no lockfile write**.
- **`pluginCacheDir()`** (`main.cpp:146-161`): `%LOCALAPPDATA%\polyglot\plugins` /
  `$XDG_DATA_HOME|~/.local/share/polyglot/plugins` / `~/Library/Application Support/polyglot/plugins`.
  (The design doc's `%APPDATA%` + `registry.json` global index were never built — see D3.)
- **Hardcoded target sites** (the fallbacks issue #30 wants gone):

  | Site | Hardcoded |
  |---|---|
  | `main.cpp:535-537` (`runBuild`) | default pair `{csharp ".cs", typescript ".ts"}` (literal extensions) |
  | `main.cpp:388-389` (`watchBuildOnce`) | checkOnly → `{"csharp"}` |
  | `main.cpp:395` (`watchBuildOnce`) | fallback `{"csharp","typescript"}` |
  | `main.cpp:676` (`runCheck`) | reference target `findTarget("csharp")` |
  | `main.cpp:930-931` (`LspServer::contextFor`) | fallback `{"csharp","typescript"}` |
  | `MSBuild.targets:69` | `--target csharp` (follow-up scope, §4) |

- **No networking, gzip/inflate, tar, or hash code exists anywhere in src/** — the in-exe downloader is
  greenfield. Reusable building blocks: the JSON reader/writer (`mintplayer/polyglot/json.hpp`),
  `readFile`/`writeFile`, `loadPluginFile`, `pluginCacheDir()`.
- **Published plugin tarball** = one file at `package/polyglot-plugin.json` (+ `package.json`); names
  `@mintplayer/polyglot-target-<t>`; versions stamped at release, **lockstep with the CLI** (P24,
  `release.yml` `plugins:` job) — a downloaded plugin at tag `X` is byte-equal to the one bundled in the
  CLI release archive `X`. The manifest itself carries **no version field**; the version travels in
  `package.json`/the packument.
- **Tests:** `main.cpp`'s CLI-level flows (pgconfig, resolution, install) are in anonymous namespaces and
  not linked into the test exe — nothing unit-tests them, and no script gate drives install/download.

## 3. Design

### D1 — Download mechanism: in-exe npm registry HTTP client (not an npm shell-out)

The flow `resolve → download → verify → extract → validate → cache → register` moves **into the exe**,
using the npm registry HTTP API directly, data-only:

1. **Resolve:** fetch the abbreviated packument — `GET <registry>/<name>` with the scope slash encoded
   (`@mintplayer%2Fpolyglot-target-python`) and header `Accept: application/vnd.npm.install-v1+json` —
   and pick the **highest version satisfying the range** (node-semver `maxSatisfying` semantics, §D4).
2. **Download** `dist.tarball` fully into memory (plugin packages are KB-scale).
3. **Verify** `dist.integrity` (SRI, `sha512-<base64>` over the raw `.tgz` bytes) **before** touching disk;
   fall back to legacy `dist.shasum` (sha1 hex) only when `integrity` is absent.
4. **Extract** in-exe: gunzip (vendored single-file inflate) + a hand-rolled ustar+pax tar reader,
   **zip-slip-safe** (reject absolute paths, `..` segments, links), stripping the `package/` prefix.
5. **Validate** the manifest with the existing `validateBackend` (full anti-silent-drop coverage
   contract) — an invalid plugin is never cached.
6. **Cache + register** (D3), then continue target resolution.

No npm lifecycle scripts can ever run — the toolchain consumes registry data only (§6 trust model).

**HTTPS transport, per platform** (the only genuinely platform-specific piece):
- **Windows:** WinHTTP (OS TLS + cert store + proxy support), ~80 lines.
- **macOS:** system libcurl (ships with the OS; headers in the CLT SDK) — link `-lcurl`.
- **Linux:** `dlopen("libcurl.so.4")` at runtime (zero link-time/runtime dependency; present on
  virtually every desktop/server distro), falling back to a **subprocess `curl --fail -sSL`**, and finally
  a clear diagnostic ("install libcurl or use a `file:` dependency / pre-warmed cache"). We do **not**
  vendor a TLS stack (mbedTLS/BoringSSL) in this milestone — recorded as the fork to revisit (PRD-§4.3
  style) if the dlopen+subprocess ladder proves painful on real machines.

**Rejected alternative (design-it-twice):** extend `runInstall`'s `npm pack` shell-out to build time.
Cheapest patch, but it makes every consumer's build depend on a Node+npm install (including `dotnet build`
of a NuGet consumer that declares a non-bundled target), contradicts the §6.1 registry-HTTP-data-only
promise, can't verify integrity before npm's own machinery has already run, and inherits npm's own config
surface. That's the workaround; the in-exe client is the principled fix. The existing `npm`/`tar`
shell-outs are **deleted** in the same change (clean cutover, no compat shim).

### D2 — `pgconfig.lock.json`: integrity pinning + deterministic offline builds

A lockfile next to `pgconfig.json`, following the package-lock v3 precedent, flat (no dependency tree —
plugins have no transitive deps):

```json
{
  "lockfileVersion": 1,
  "packages": {
    "@mintplayer/polyglot-target-python": {
      "version": "0.3.1",
      "resolved": "https://registry.npmjs.org/@mintplayer/polyglot-target-python/-/polyglot-target-python-0.3.1.tgz",
      "integrity": "sha512-…"
    }
  }
}
```

Resolution is **lock-first**: if the locked `version` satisfies the requested range and the cache holds
that exact entry (integrity re-verified, D3), **no network I/O happens at all** — builds are deterministic
and offline-capable. The registry is consulted only when (a) no lock entry exists, (b) the lock no longer
satisfies a changed range, or (c) `polyglot install --update` explicitly re-resolves. Every successful
download writes/updates the lock entry. The lockfile is meant to be committed.

### D3 — Versioned shared cache; the §6.1 `registry.json` global index is superseded

Cache re-keyed from bare target name to **npm name + resolved version**:

```
<pluginCacheDir>/@mintplayer/polyglot-target-python/0.3.1/polyglot-plugin.json
                                                          /meta.json
```

`meta.json` records `{ name, version, resolved, integrity, manifestSha512 }`; on every load the cached
manifest is re-hashed against `manifestSha512` and the lock's `integrity` is matched against `meta.json` —
a tampered or torn cache entry is refused and re-fetched (the §6.3 "re-verified on load" promise, made
concrete). Writes are atomic (temp dir + rename) so concurrent builds can't observe a half-written entry.

**Decision:** the global `registry.json` index promised in §6.1 is **not built**. The per-project lockfile
pins versions+integrity, and the versioned cache directory *is* the machine-wide installed set — a separate
index would be a second source of truth to keep consistent for no consumer. `plugins-and-targets.md` §6.1
is updated to record this supersession (and the cache-path discrepancy: code's `%LOCALAPPDATA%`/
`$XDG_DATA_HOME` locations are the reality; the doc's `%APPDATA%` was never built).

### D4 — Dependency spec grammar + semver subset

`dependencies` values accept: `file:<dir>` (unchanged, never locked — local path), exact `1.2.3`,
`^1.2.3`, `~1.2.3`, `>=1.2.3`, and `latest` (resolved via `dist-tags.latest`). Keys are npm package names;
a bare name normalizes to `@mintplayer/polyglot-target-<name>` (the existing `runInstall` convention,
`main.cpp:1374-1375`). The plugin's **target name** comes from the downloaded manifest's `name` field —
`targets` entries are matched against loaded manifests, never against package names.

Semver implementation is the minimal correct subset: parse `(major,minor,patch,prerelease[])`; desugar
`^`/`~` per node-semver (`^1.2.3`→`>=1.2.3 <2.0.0-0`, `^0.2.3`→`>=0.2.3 <0.3.0-0`,
`^0.0.3`→`>=0.0.3 <0.0.4-0`; `~1.2.3`→`>=1.2.3 <1.3.0-0`); pick the highest satisfying version;
**prerelease versions are excluded** unless the range names that exact `[major.minor.patch]`-prerelease.

**In-box-vs-range rule:** an in-box plugin's version **is the CLI's own version** (the P24 lockstep
invariant — no manifest change needed). A dependency on a bundled target uses the in-box copy when the
CLI version satisfies the range, or when the CLI is a dev build (`0.0.0-dev` — dev convenience); otherwise
the requested version is fetched and **shadows** the in-box registration for that build.

### D5 — Remove the hardcoded fallbacks: the plugin set is fully config-sourced

- **`runBuild` no-config default pair** (`main.cpp:535-537`) — deleted. No `pgconfig.json` found **and**
  no `--target` → a clear diagnostic listing the loaded targets and suggesting `--target <t>` or a
  `pgconfig.json`. (Behavior change, accepted: quick one-offs say `--target csharp`; projects declare
  `targets`.) Output extensions come from `Backend::fileExtension()` everywhere — the literal
  `".cs"`/`".ts"` pair dies with the fallback.
- **watch fallback** (`main.cpp:395`) — same rule as `runBuild`.
- **`check` reference target** (`main.cpp:676`) and watch checkOnly (`main.cpp:388-389`) — the fixed
  `"csharp"` is replaced by a derived `referenceTarget()`: the first loaded backend whose §3.A coverage/
  capability surface is complete (today that *derives to* C#, but by capability, not by name — a
  hard-bound name is exactly what #30 forbids).
- **LSP fallback** (`main.cpp:930-931`) — with a pgconfig, `pc.targets` (as today); without one the LSP
  must still analyze: diagnostics run against `referenceTarget()` and per-target reserved-name checks run
  against **all loaded** targets (superset = safe squiggles, no invented pair).
- `runInstall`'s `@mintplayer/polyglot-target-<name>` normalization stays — it's a naming convention, not
  a target list.

**LSP latency/failure containment:** resolution results (including failures) are memoized per config
generation in-process, so a dead network costs one attempt, not one per keystroke; with a satisfied lock +
warm cache the LSP path stays network-free. A resolution failure surfaces as a diagnostic, never a hang.

### D6 — Registry/proxy configuration (minimal respectable subset)

Honor `.npmrc` `registry=` and `@scope:registry=` keys (project dir upward, then `~/.npmrc`), the
`HTTPS_PROXY`/`https_proxy`/`NO_PROXY` environment variables (WinHTTP additionally picks up system proxy
settings), and a `POLYGLOT_REGISTRY` env override (used by the test harness's fake registry). **No auth
tokens** — public packages only, documented as unsupported (a private-feed story is future work).

### `polyglot install` after P30

Remains, redefined over the same in-exe pipeline (the `npm pack`/`tar` shell-outs are deleted):
pre-warms the shared cache (CI, offline prep) and, when run inside a project, writes/updates the lock.
`polyglot install --update` re-resolves ranges past the lock. The bare-name→cache-probe path
(`pluginCacheDir()/t/…`) dies: a `targets` entry not covered by in-box or `dependencies` is a diagnostic
telling the user to add the dependency — everything sourced from `pgconfig.json`.

## 4. Scope contract

**In scope:** the in-exe npm fetch pipeline (D1); lockfile (D2); versioned cache + on-load re-verification
(D3); semver subset + spec grammar (D4); removal of all CLI hardcoded target fallbacks (D5); registry/proxy
config subset (D6); `install` rewrite; the fake-registry script gate; doc updates (§6/§6.1/§6.3,
json-plugins §5.4, PRD §4.20 design record).

**Follow-up-gated (issue #30 marks it "possibly its own issue"):** `MintPlayer.Polyglot.MSBuild` dropping
`--target csharp` for pgconfig-driven multi-target emission. It needs its own design (where non-`.cs`
outputs land in a csproj build, an `outputs` mapping in pgconfig, preserving the P28 single-Exec/
single-stamp incrementality). P30 keeps the `.targets` behavior byte-identical; the CLI work makes the
follow-up possible. Sketch in PLAN.md slice 7.

**Out of scope:** `environments` gating (§6 — parsed nowhere today, separate feature); auth/private
registries; a vendored TLS stack on Linux (recorded fork); generic URL/file-hosting feeds (§6.1 fallback
feed — the npm feed is the shipped one); plugin signing beyond SRI integrity.

## 5. Failure modes & diagnostics (never a miscompile, never a hang)

| Condition | Behavior |
|---|---|
| Offline + no lock entry (or lock unsatisfied) | error naming the package, the range, and the fix (`polyglot install`, network, or `file:`) |
| Offline + lock satisfied + cache warm | **builds normally, zero network** |
| Integrity mismatch (download) | refuse, delete nothing from cache, name expected/actual algorithm+hash |
| Tampered/torn cache entry | refuse that entry, re-fetch when online, diagnostic when not |
| Tarball path traversal / links / oversized | refuse extraction (zip-slip guard), name the entry |
| `validateBackend` failure | refuse to cache, print the validation error (existing behavior) |
| Unknown package / no version satisfies range | error with the range and the available versions (from packument) |
| No transport available (Linux, no libcurl/curl) | error explaining the ladder and the `file:`/pre-warm alternatives |

## 6. Security / trust model (unchanged, now enforced)

Downloaded plugins are **data, not code** — JSON manifests validated by `buildBackend`; no lifecycle
scripts, no executable bits, extraction is path-traversal-safe, parsing is bounded (the existing JSON
reader + a size cap on tarball/inflated payloads against zip-bombs), integrity pinned in a committed
lockfile and re-verified on load. This implements the §6 trust-model checklist verbatim.

## 7. Test & gate plan

- **Unit** (new translation units are testable — not anonymous-namespace `main.cpp` internals): semver
  table tests (node-semver-derived fixtures incl. prerelease exclusion); SRI verify (sha512/sha1, mismatch);
  gzip inflate + tar reader over checked-in golden `.tgz` fixtures (incl. pax long-path entry, a
  traversal/zip-slip fixture that must refuse); lockfile read/write round-trip; spec-grammar parsing.
- **Script gate** (wired into `scripts/build-and-test.ps1`): a **fake registry** — a tiny Node HTTP server
  (Node is already a harness dependency) serving packument + tarball fixtures built from the repo's own
  `plugins/` — pointed at via `POLYGLOT_REGISTRY`. End-to-end: empty cache + `pgconfig.json` declaring a
  python dependency → `polyglot build` fetches, locks, emits, conformance-diffs; server killed → second
  build succeeds **offline** from lock+cache; tampered tarball and tampered cache entry → refused with the
  right diagnostics; `--target` override still wins.
- **Existing gates stay green byte-identical:** unit suite, differential C#/TS/Python conformance,
  MSBuild/NuGet script test (unchanged `.targets`), watch protocol, buildfile-parity (new sources added to
  both `.vcxproj` and `CMakeLists.txt`).

## 8. Blast radius & versioning

- Projects using in-box targets or `file:` deps: unchanged.
- Manually-installed-cache users: the bare-name cache probe dies — add the `dependencies` entry (clean
  cutover, per project convention; the diagnostic says exactly this).
- Bare `polyglot build foo.pg` with no config and no `--target`: now errors (was: silent csharp+ts pair).
- NuGet/MSBuild consumers: byte-identical behavior (`--target csharp` untouched until the follow-up).
- VS Code extension: no change required (LSP path gains auto-download transparently; lock+cache keep it
  network-free in steady state).
- **Versioning:** lockstep, tag-driven (P24) — no source bump; the PR carries `release:minor`. One PR,
  slices as commits.
